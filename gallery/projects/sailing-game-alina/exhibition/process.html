Development Process

A chronological record of human-AI collaboration on this project.

Project Setup

Student: Alina Zaigralova

Domain: Sailing

AI Agent: WorkBuddy

Core Learning Shift: Beginners think sailing means steering toward the target. Experts know sailing means reading the wind and adjusting the boat, sail, and route in relation to it.

Technical Track: Track A — No build step (HTML, CSS, plain JavaScript, Canvas 2D)

Development Timeline

Session 1— Planning & Setup

Brief Analysis & Implementation Plan

Read the development brief and identified the core learning shift (wind reading vs. target steering). Chose Track A (plain HTML/CSS/JS + Canvas 2D) because the game is 2D and this track requires no build step — ideal for the exhibition environment on an iMac.

Defined three challenge presets that change system variables: Downwind Run (wind behind), Beam Reach (wind perpendicular), and Upwind Challenge (target upwind, requiring tacking).

Session 1— Core Implementation

Sailing Physics Engine

Built the core game engine in assets/game.js. Implemented simplified sailing physics: angle of attack between wind direction and boat heading determines base speed via a polar curve (peak at beam reach, zero in the no-go zone). Sail trim efficiency is calculated by comparing the player's sail angle to the optimal angle for the current heading.

Environment variables (wind, current, rocks, target, time) are clearly separated from player-controlled variables (heading, sail angle) and calculated results (speed, drift, stability, distance) in the code, as specified in the brief.

Session 1— Visual Feedback

Feedback System

Implemented all feedback channels from the brief: sail flapping animation (inefficient angle), wake trail length (speed), wind direction arrow (environment reading), collision warning rings on rocks, HUD bars for speed/trim/stability, and text messages ("Good trim!", "Sail is flapping!", "Watch out for rocks!").

Added a "point of sail" indicator (In Irons, Close-Hauled, Beam Reach, Broad Reach, Running) to help players connect the visual experience with sailing terminology.

Session 1— Website

Project Website

Created the integrated project website with four pages: Home (game idea, domain knowledge, system design), Play Game (hosts the canvas), and Development Process (this page). Shared stylesheet provides coherent styling. All links use relative paths for Gallery compatibility.

Session 1— Testing

Initial Testing

Tested all three challenges: navigation links, game start, steering, sail trim, win/lose conditions, and restart. Verified the complete observe → judge → act → feedback → adjust loop works in each challenge.

How AI Helped

The AI agent (WorkBuddy) was responsible for: reading and interpreting the development brief, choosing the technical approach (Track A), implementing the full game engine and website, creating the project file structure, and maintaining the development log. The student provided the domain choice (sailing), the design direction (adventure, discovery), and the core learning shift.

Relationship to the Core Learning Shift

The game is designed so that the Upwind Challenge (Challenge 3) directly confronts the player with the novice misconception. The target is directly upwind. A player who simply steers toward it will enter the no-go zone, their sail will flap, and the boat will stop. The only solution is to tack — sail at ~45 degrees to the wind in one direction, then turn and sail at ~45 degrees in the other. This is the expert behavior the game aims to teach.

The first two challenges (Downwind Run, Beam Reach) build the skills needed: reading the wind indicator, trimming the sail, and understanding how speed responds to angle changes. By the time the player reaches Challenge 3, they have the tools to discover tacking through experimentation.
