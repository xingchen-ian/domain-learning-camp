Development Process

A chronological timeline of how this project was built through human-AI collaboration. This timeline is based on agent-development-log.md and reflects the actual development path — including mistakes and unfinished ideas.

Dependency Track: Track A — No build step (HTML, CSS, Canvas 2D, plain JavaScript)

Development Timeline

Session 1 — Project Initialization

📄 Brief Read & Project Structure Created

The AI Agent read the full development brief and the domain-learning design summary. The brief specified a baking-themed game where the core learning shift is moving from "just follow the recipe" to "check materials, understand equipment, bake carefully."

The project directory structure was created following the expected file layout: exhibition/ for the final static website, source/ for development source (not needed for Track A), and supporting documentation files.

AI Setup Structure
Session 1 — Game Design

🎮 Game Loop & Challenge Presets Designed

Based on the brief's core loop (Observe → Judge → Act → Feedback → Adjust), the game was designed with four phases: Menu (recipe selection), Pantry (ingredient selection), Mixing (timing mini-game), and Baking (oven monitoring). Each phase maps to a real baking skill.

Four challenge presets were defined — Cookies, Bread, Brownies, and Cake — each with different variables: different correct/wrong ingredients, different mix-time windows, different bake-time windows, and different difficulty levels. The key learning difference: cookies use baking soda, bread uses yeast, brownies need no leavening, and cake uses baking powder.

AI Design Game Loop
Session 1 — Implementation

💻 Game Built with Canvas 2D

The complete game was implemented in a single game.js file using HTML5 Canvas 2D — no external libraries, no build step. Track A was chosen because the game is 2D and simple graphics are sufficient per the brief's guidance to "prioritize interaction, feedback, and learning over polish."

The game code separates environment data (recipe definitions, oven state), player-controlled data (selected ingredients, mixer actions), and calculated results (scores) into clearly labeled sections, as required by the brief.

AI Implementation Canvas 2D
Session 1 — Website

🌐 Website Pages Created

Three HTML pages were built: index.html (home with game idea, domain knowledge, and system design sections), game.html (playable game page with instructions), and process.html (this development timeline page). A shared CSS stylesheet provides the warm baking-themed design.

The website includes a system graph diagram (SVG), data model blocks, a comparison table of novice vs. expert thinking, and a challenge preset table showing how each recipe changes variables. Navigation is consistent across all pages.

AI Implementation Website
Session 1 — Testing

🧪 Local Testing & Feedback

The game and website were tested through a local static HTTP server. Navigation links between all pages were verified. The game's full loop was tested: menu → pantry → mixing → baking → result → retry. Canvas rendering was verified at 1920×1080 exhibition resolution.

The game was confirmed to run without internet access, with all assets (CSS, JS) stored locally. No external dependencies, CDNs, or build tools are required for the exhibition version.

AI Influence on Design Direction

🤖 What the AI Agent Did

The AI Agent read the brief, designed the game loop mapping to the observe-judge-act-feedback-adjust cycle, chose Track A (no build step) as the simplest viable dependency track, designed the four challenge presets with different variables, implemented the complete Canvas 2D game, built the three website pages with a system graph and data model, and created all supporting documentation.

👨‍🎓 Student Decisions

The student provided the core design brief: the baking domain, the novice misconception (just follow a recipe), the expert approach (check materials, use equipment, bake carefully), and the success condition (bake 4 items). These student-authored decisions drove all AI implementation. The student should review the game and provide feedback on whether the experience matches the intended learning shift.
