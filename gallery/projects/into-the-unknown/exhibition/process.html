Development Process

A chronological timeline of how the game was built through human-AI collaboration.

Development Overview

Student / Team: Hawinate
AI Agent: WorkBuddy
Domain: Cave Exploration
Dependency Track: Track A — No build step (HTML, CSS, plain JavaScript, Canvas 2D)
Core Learning Shift: From exploring by instinct → making careful decisions and resource management

Development Timeline

Session 1 — Planning
Reading the brief and understanding the domain
The development brief was provided as a Markdown file. The AI Agent read the full brief, including the design summary, core learning loop, data model, rules, feedback priorities, and technical requirements. The core learning shift (instinct → careful decisions) was identified as the guiding principle for all gameplay decisions.
Session 1 — Architecture
Choosing the technology track
Based on the brief's guidance to "choose the lowest necessary dependency track," Track A (no build step) was selected: plain HTML, CSS, and JavaScript with Canvas 2D. This ensures the exhibition build runs on any static HTTP server without npm, Node.js, or a development server. All code is self-contained with no external dependencies.
Session 2 — Game Engine
Building the core game engine
The game engine was implemented in a single JavaScript file (game.js) using Canvas 2D. Key systems built: tile-based cave rendering, player movement with WASD/arrow keys, flashlight cone with directional illumination, fog-of-war darkness, battery drain system, health system, timer, collision detection, and screen shake effects. Data was separated into Environment Data, Player-Controlled Data, and System-Calculated Results as specified in the brief.
Session 2 — Level Design
Designing three challenge presets
Three levels were designed as grid-based maps, each changing system variables and relationships: Level 1 (The Entrance) teaches basics with bats only; Level 2 (The Depths) adds water and trap hazards plus the rescue objective; Level 3 (The Escape) adds unstable ceiling collapse events and tighter time/battery constraints. Footprints were placed on safer paths to reward observation.
Session 2 — Feedback Systems
Implementing feedback translation
Invisible game data was translated into readable feedback: flashlight flickering when battery is low, screen shake and dust particles for collapse warnings, footprint marks on the floor as path clues, a red vignette overlay when danger level is critical, and a Web Audio API sound system for pickup, hurt, bat, splash, rumble, and victory effects.
Session 3 — Website
Building the integrated project website
Four HTML pages were created with shared CSS: Home (game idea), Domain & Design (domain knowledge, system design, feedback mappings, challenge presets), Process (this timeline), and Play Game (the playable game embedded in the page). Navigation is consistent across all pages, and all links use relative paths for Gallery compatibility.
Session 3 — Testing
Testing and verification
The game was tested through a local static HTTP server. Navigation links, gameplay controls, all three levels, win/lose conditions, and restart functionality were verified. The layout was checked at 1920×1080. All assets are local — no internet access required.

Key Design Decisions

1. Flashlight as the Core Resource

The flashlight is both the player's primary tool and their most limited resource. It drains faster when moving, encouraging the player to think about whether exploration is worth the battery cost. When it runs out, the player is nearly blind — directly teaching the consequence of poor resource management.

2. Footprints as Teaching Feedback

Footprints appear on the safer path, rewarding players who observe their surroundings. This directly addresses the novice misconception of "rushing forward without carefully observing." A player who follows footprints succeeds; one who ignores them may hit hazards.

3. Progressive Hazard Introduction

Level 1 introduces only bats (low damage, easy to understand). Level 2 adds water and traps (medium-high damage, requiring caution). Level 3 adds unstable ceilings with periodic collapse warnings (environmental pressure). This progression teaches the player to predict dangers through repetition.

4. Visible Feedback Over Numbers

Per the brief's feedback priorities, invisible data is shown through visual and audio effects rather than just numbers: flickering light instead of just a battery percentage, screen shake instead of a "danger score," footprint marks instead of a directional arrow. The HUD bars exist for quick reference, but the primary feedback is experiential.

AI Influence on Design Direction

The AI Agent (WorkBuddy) contributed the following design directions, all of which were accepted by the student:

  • Track A technology choice: Recommended plain HTML/CSS/JS over npm/Vite/Three.js to keep the exhibition build simple and Gallery-compatible.
  • Flashlight cone mechanic: Suggested a directional cone of light rather than a circular radius, making the facing direction meaningful and increasing tension.
  • Three-level progression: Proposed gradually introducing hazards (bats → water/traps → collapse) rather than all at once, matching the learning loop.
  • Web Audio API for sound: Suggested generating sounds programmatically instead of loading audio files, keeping the build self-contained.
  • Red vignette for danger: Proposed a visual danger indicator that appears when health/battery/time is critically low, rather than relying only on HUD numbers.
