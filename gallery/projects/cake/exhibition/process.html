Development Process

A concise, truthful timeline of how the project was built, including the changes, tests, and decisions along the way. The full log lives in agent-development-log.md at the project root.

Timeline

1. Read the brief and the system graph

Stage 1 · Planning

Read development-brief.md and looked at the system graph. Identified the core learning shift, the player loop, and the three difficulty presets. The graph specified the controls (A, W, mouse, space) and the win / lose conditions, which I used as the spine of the implementation.

2. Chose the dependency track

Stage 1 · Decision

Picked Track A — no build step. The brief allows Track C (Vite + Three.js) for 3D, but a 2D Canvas game does not need that complexity. Track A means the iMac just needs a static HTTP server, no npm install and no internet.

3. Built the smallest complete game loop first

Stage 2 · Implementation

Built a single-state game object that holds the order, the cake, the step, and the timer. Wrote the order generator, the action handlers, the canvas drawing, and the win / lose check. Three levels were then layered on top as different configurations of the same loop.

4. Built the integrated project website

Stage 2 · Implementation

Created the four required pages — Home, Domain & System, Development Process, and Play the Game — with shared navigation, Gallery-compatible relative paths, and a 1920×1080 layout. Linked the website to the brief and the system graph.

5. Tested the build

Stage 3 · Verification

Started a local static HTTP server, opened exhibition/index.html, and walked through all four pages, the level select, and the three levels. No console errors. All links and assets use relative paths so the project will work inside gallery/projects/group-XX/.

6. Packaged the submission

Stage 3 · Submission

Created README.md, THIRD_PARTY.md, submission-manifest.json, and preview.png. The final ZIP is named 2026-Camp-Group-XX-Cake.zip and opens directly to the project files.

What changed along the way

  • Decoration toggle. The first version made decorations add-only. The brief says players should be able to replace a wrong decoration. Decorations are now toggled on click, and the submit button is disabled until at least one decoration is added.
  • Game Over on timeout. When the timer reaches zero, the game immediately shows "Game Over" and freezes — no cake evaluation, no continuing. The player lost because they ran out of time.
  • Color-coded timer. The timer turns yellow with 20 seconds left and red with 5 seconds left. This is a small but real translation of invisible data ("time left") into visible state.
  • Timed mixing and baking phases. Mixing the batter and baking in the oven each take 10 seconds with a visible progress bar. The player cannot act during these phases. This adds realistic tension and forces careful planning before hitting Mix or W.
  • Level 3 reworked: layer selection. Level 3 now has a 1-minute timer (changed from 30 seconds). The key twist: the player must manually pick a different base for each of the 3 layers. Step-by-step instructions and a "memorize the toppings" warning were added to the order panel.
  • Ingredient click fix. The original ingredient click handler depended on the global event object, which silently failed. Fixed by passing the button element directly to the selection function.
  • Decoration phase: no hints. During the DECORATE step, the order card hides the "Toppings" list and shows "Remember from the order!" instead. This forces the player to recall what decorations the customer asked for rather than just copying from the order card.

Open questions and known issues

  • The cake drawing uses simple Canvas 2D shapes. It is intentionally not photoreal. The brief says to keep graphics simple and prioritize interaction over polish.
  • Level 2 and Level 3 use the same number of decorations as the brief calls out. The student can adjust the constants minDecorations and maxDecorations in game.js to change the difficulty.
