Timeline

From brief to first playable, through a playtest

2026-07-21 · Planning

Read the brief, lock identity

Read the full development brief (sections 1–13) and confirmed two things with the student: the project is called First Diagnosis, and there is no system graph — the brief is the sole primary reference. This decision is documented in the agent log so the website is honest about what the project was and was not.

2026-07-21 · Tech decision

Picked the lowest necessary dependency track

The brief gives three tracks. Track A (HTML + CSS + plain JS + Canvas 2D) was chosen because the design is 2D, the audience is high-school students, and the iMac exhibition is simpler without a build step. No npm, no Vite, no Three.js. The chosen track is documented in the README.

2026-07-21 · Data model

Designed the smallest complete loop

The agent proposed: one patient, one cut, one hidden variable (cleanliness), four actions (examine, clean, stitch, bandage), one visible body signal (bleeding), one delayed signal (stitch pops if not cleaned). This is the smallest loop that exercises the "evaluate before treating" learning shift.

2026-07-21 · Build (v1)

Three challenge presets, four-page website, audit pass

The first playable shipped with three challenges — The First Cut (dirty cut, basic loop), The Deep Gash (depth is hidden, basic stitch is not enough), The Limp Arm (no x-ray, conservative care on a broken bone). Each one changes which variables matter and which equipment is available. The four pages (Home, Domain & System, Development, Play) link with relative paths. Audit pass: no node_modules, no external assets, 1920×1080 capture saved to preview.png.

2026-07-21 · Refinements

Three small decisions locked

The student locked three open items: no team name in the footer, audio on by default, and the deep-cut tool is called Advanced suture. All three applied across the codebase and the dev log.

2026-07-21 · Self Play Test 01

The single biggest design change so far

The student played the v1 build and surfaced the core gap: the treatment view is a single screen, but a real disaster has multiple patients, scattered equipment, and a per-patient clock that the world keeps running while the player focuses on one of them. The v1 design didn't have anywhere for that to live.

2026-07-21 · Build (v2)

Rewrote the game as exploration + treatment

The whole game was rewritten around the new design: a top-down world (forest clearing, crashed jet, fire), a moveable medic (WASD/arrows), seven patients per mission (4 visible, 3 hidden), per-patient 170 s timers that start on find and keep ticking during treatment, sidebar equipment (saline, small bandage, suture, medicine) and environment equipment (cast, big bandage, cleaning cloth, water — water required to take medicine). The click-to-select-injury bug was fixed (50 px hit radius, dedicated treatment-mode click handler, injury IDs auto-assigned at level start). The world pauses its per-patient stability drain only while the action bar is mid-action; the world and other patients' clocks keep running. The website shell was rebuilt to match.

Pending · Audit

Submission audit before packaging

The brief's §11 audit will run before packaging: no node_modules, relative paths only, all links work from gallery/projects/group-XX/, the game runs at 1920×1080, the manifest is filled in honestly, and the public-display consent is confirmed with the student.

Decisions and where they came from

What the AI suggested vs. what was decided

Per the brief, the development log records who decided what. A short version of that record is below.

Project title

The student chose First Diagnosis. The AI did not propose alternatives.

System graph

The student said there is no graph and to ignore that part. The AI is working from the brief only.

Tech track

Track A was recommended by the AI following the brief's "lowest necessary dependency" rule. No npm, no Three.js.

Mission structure (v2)

The student requested a single-mission design with 5–7 patients, an explorable environment, and equipment that has to be found. The AI implemented this as one mission (Jet Crash — Forest Clearing) with 7 patients and 7 env equipment spawns.

Per-patient timer

The student requested 160–180 s per patient, clock starts on find. The AI used 170 s (mid-point), keeps ticking during treatment so the world pressure is real.

Equipment split

The student split equipment into "always in the sidebar" (saline, small bandage, suture, medicine) and "must be found" (cast, big bandage, cleaning cloth, water). The AI implemented this exactly; water is required to take medicine.

Movement

The student requested a moveable character. The AI added WASD/arrow movement with circle-vs-rect collision against the jet, trees, and fire patches.

Audio

Light Web Audio synth for find chime, stitch pop, success / fail chord, pickup, warn. On by default per the student. No external audio assets.

Visual style

Soft clinical palette for the website (blue / white / red accents). A darker, more atmospheric palette for the in-game world (forest green, jet gray, fire orange). The brief explicitly asks for readability over polish, so this is intentional.
