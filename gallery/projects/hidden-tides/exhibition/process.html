Development Process

Timeline

Phase 1 — First Build (Interaction 01–04)

Built the complete game from scratch based on the development brief:

  • game.js + game.html: Canvas 2D sailing survival game with all core mechanics
  • 3 challenge presets: Calm Passage, Shark Waters, Storm Run — differing by system variables
  • Website pages: index.html (home), domain.html (system design), process.html (timeline)
  • Documentation: README.md, THIRD_PARTY.md, agent-development-log.md
  • Track A: Pure HTML/CSS/JS with no build step or external dependencies

Phase 2 — Keyboard Input Debugging (Interaction 05–08)

The player reported that WASD keys had no response in the browser. Multiple debugging rounds were conducted:

  • Issue 1: Game started in menu state; player needed to click a difficulty button first. Added auto-start on WASD press from menu.
  • Issue 2: macOS Chinese IME was intercepting letter keys before they reached the game. Switched from e.key to e.code (physical key position) for input.
  • Issue 3: Overlay div was not fully releasing keyboard events. Added pointer-events:none and innerHTML clear on hide.
  • Issue 4: Canvas lacked focus. Added tabindex and canvas.focus() on game start.
  • Issue 5: The !env condition in auto-start logic blocked triggering in some edge cases.
  • Diagnostic tools added: HTML-level key event monitor (yellow bar), per-frame state/velocity display, boat color change on keypress.

Root cause identified: Multiple layers of issues compounded — IME interception, overlay event blocking, focus management, and conditional logic bugs. Each layer masked the others, making diagnosis difficult.

Phase 3 — Complete Rewrite (Interaction 09)

Due to persistent keyboard input issues across multiple fix attempts, a full rewrite was performed with these design principles:

  • Simpler input architecture: Single keyDown{} object, populated by both e.code and e.key on every keydown event
  • isDown() helper: Checks all possible key identifiers at once (e.g., ["KeyW","w","ArrowUp"])
  • All game keys get preventDefault(): No exceptions for WASD
  • No complex auto-start conditions: Clean button-only start from menu
  • Built-in diagnostic display: Bottom-right HUD always shows active keys and velocity when playing

Key Technical Decisions

  • Canvas 2D over WebGL/Three.js: This is a 2D top-down sailing game. Canvas 2D is sufficient and keeps dependencies at zero (Track A).
  • Web Audio API for sound: All sounds generated procedurally — no audio files needed. Works offline.
  • IIFE pattern: Game code wrapped in an IIFE to avoid polluting global scope.
  • 1280x720 virtual resolution: Scales to any screen via CSS while keeping consistent game coordinates.
  • Shark placement algorithm: Sharks are deliberately placed far from rocks (180px minimum), encoding the "open water = danger" rule into level generation itself rather than just gameplay mechanics.

Known Issues & Future Work

  • Keyboard input reliability on macOS with CJK input methods requires careful handling of both e.code and e.key
  • Pink zone "angel" visuals are text-based (✡ symbols); could be improved with simple sprite drawings
  • No touch/mobile controls yet — designed for desktop browser with keyboard
  • preview.png and submission-manifest.json need to be generated before final submission
