Development Process

A truthful timeline of how this project was built, including the decisions, the dead ends, and the things that are still rough.

Timeline

Stage 1 - Read the Brief
Read the design brief and the system graph
Read An-adventure-on-horseback.md in full. Identified the three challenge presets (simple / complicated / roundabout), the three win-lose conditions, and the horse emotion system. Confirmed that the student has 4 years of real riding experience and wants the game to feel like real riding, not a cartoon.
Stage 2 - Choose a Stack
Pick Track A: HTML + CSS + Canvas 2D + plain JS
The brief asks for the lowest necessary dependency. Three.js would have looked pretty but the iMac in the gallery does not run a build pipeline, so a single static folder is the right answer. Canvas 2D is enough to draw a 3/4-perspective scene with a detailed horse.
Stage 3 - First Game Loop
Build the smallest complete loop: speed, line, take-off, jump
Wrote game.js with a single state object (Game) that holds environment data, player input and calculated results as separate fields. The first playable loop is: press W to accelerate, steer with A/D, press Space near an obstacle to jump. If the take-off point is in the window and the horse is willing, the bar stays up; otherwise the bar falls and the mood drops.
Stage 4 - Three Levels
Add Level 2 (S-curve, 90s) and Level 3 (roundabout, 75s, spirited horse)
Each level is a data object in the LEVELS array: route polyline, obstacle list, time limit, horse config. The render code does not know which level it is on; the data drives the experience. This is the part the brief explicitly asked for: "challenge presets should change system variables or relationships, not only visual decoration."
Stage 5 - The Horse, Not a Cartoon
Redraw the horse with realistic anatomy and a rider
First pass was a simple ellipse horse with stick legs. It looked like a children's game. Replaced with a layered body (chest, barrel, hindquarters), proper leg joints, mane, tail with strands that move with the wind, a saddle, a rider in boots and a helmet. Added a sweat-droplet effect for tired state and red-eye effect for angry state. Still 2D, but reads as a real horse.
Stage 6 - The Horse's Mood
Wire the emotion system so the player can feel it
Added a 0-100 emotion value with four bands: happy, calm, tired, angry. The HUD shows the band name and a coloured bar. The body of the horse shows sweat drops when tired and red eyes when angry. When the emotion drops below 12, the horse randomly refuses to move. Whip drops emotion, soothe raises it. The student's core insight ("a horse is an organism with its own feelings") is now a live system.
Stage 7 - Website and Process
Build the four required website pages
index.html for Home, system.html for the system graph and feedback table, process.html for the timeline you are reading, and game.html for the playable build. All navigation is relative so the folder can be dropped into gallery/projects/group-XX/ without rewriting paths.
Stage 8 - Things That Are Still Rough
Honest list of unfinished work
The 3/4 perspective uses a single vanishing point and a flat road, not a true 3D engine. Trees in the distance are procedurally generated ellipses. There is no real audio. The third-person camera does not bank into turns as a movie camera would. These are conscious choices: the brief says to keep the first version small and prioritize the learning loop.
Stage 9 - Documentation and Audit
Create README, development log, manifest, system graph, and final ZIP
Added README.md, agent-development-log.md, submission-manifest.json, THIRD_PARTY.md, and system-graph.png. Verified every exhibition page loads from a static HTTP server, uses relative paths, makes no external requests, and fits 1920×1080. Packaged the project as 2026-Camp-Group-XX-An-adventure-on-horseback.zip.

Decisions Log

Why Canvas 2D instead of Three.js?

The gallery iMac only opens index.html directly through a static HTTP server. A Three.js project would force the gallery to install Node and run a build. Canvas 2D delivers the perspective look the student wanted without that cost. If the student wants a real 3D version later, the data model in game.js is already 3D-ready (world X, Z and a projectGround() helper), so a Three.js port is mostly a renderer swap.

Why three different horses, not one horse with different skins?

The brief says the three levels should change the relationship between the player and the system, not only the scenery. Whisper, Dapple and Storm each have a different angerRate and maxSpeed: Storm gets angry fast and goes fast, Whisper is forgiving and slower. Playing well on Storm requires actually managing the mood; playing well on Whisper does not. That difference is the point.

Why a 3/4 behind view, not a side-scroller?

A side-scroller reads as a 2D platformer. The student said "simulate a real-life horse riding" and the brief says the graphics should not be cartoon. A 3/4 behind view is the most common view in real horse-riding games and the most natural way to read the road ahead.
