Development Process

A truthful, chronological record of how Squint was built through human–AI collaboration. Full raw entries live in agent-development-log.md.

Interaction 01 — Project scaffold & first playable loop

Stage: Setup & Milestone 1 · Student: Charlie. S · Agent: Claude

Request. Student confirmed the implementation plan and asked the agent to build the project following the brief and system graph.

What changed. Created the exhibition/ + source/ structure, copied the brief and system graph, built the three website pages, and implemented a complete Level 1 loop: procedural reference scene under one light, hidden true-value map, click-drag block placement, 1–5 value assignment, squint / step-back / flip feedback, region pops, stroke budget, deterministic AI critique, and win / lose. Levels 2 and 3 were wired as selectable presets.

Decisions. Track A (no build, Canvas 2D + plain JS). The "AI critique" is a deterministic offline scorer (compares the player's blocked-in values region-by-region against the hidden true-value map) so the exhibition runs without internet. Win rule follows the system graph: all regions matched AND budget remaining AND score ≥ 75.

Test. Served locally over a static HTTP server and verified navigation, gameplay, squint/step-back/flip, scoring, and win/lose at 1920×1080. Passing score threshold (75) and stroke budgets still need playtesting by the student.

Open question for the student: are the 75 / 35 thresholds and the 14-stroke budget right once you play it?

Interaction 02 — Painting model & detail mode

Stage: Engine refinement · Student: Charlie. S · Agent: Claude

Request. The block-drag mechanic was clunky — the player "draws spheres but can only draw with shapes." Student asked for individual pixel painting, a higher pixel count, attainable detail, and an optional detail mode after completing a level.

What changed. Replaced block-drag with per-pixel painting: grid raised from 21×21 to 84×84; click / drag paints cells; an adjustable brush (single pixel up to 17×17) lets the player work from detail to big masses; stroke budget now counts one drag (rewarding efficient mass-blocking); squint "pops" recomputed on a coarse 21×21 grid for readability; and an optional Detail Mode unlocks on win for free, unscored refining.

Validation. Headless Node harness confirmed the full 1–5 value range is still used, a perfect solve scores 100 / wins, a blank canvas scores 0, and a naive one-value-per-region attempt correctly fails (a sphere has a gradient, so several values per region are required). Exhibition serves offline with no CDN / absolute paths.

Playtest the new painting feel, brush defaults, and the 20 / 24 stroke budgets.

Reflection 01 — Stage Reflection

Covers Interactions 01–02 · Stage: first playable loop → painting model refinement

Goal of this stage. One complete observe → judge → act → feedback → adjust loop that teaches the core shift (light → values, not objects).

What changed. Website + game skeleton plus a working Level 1, then a full rewrite from block-dragging to per-pixel painting with adjustable brush (84×84 grid). Detail mode unlocked on win.

How AI helped. Drafted the plan, scaffolded all files, implemented the engine, caught an inverted-quantize bug via headless validation, then rewrote the interaction model entirely after student critique about clunky drawing.

Student reflection. The game does deliver the intended learning shift. Two gaps: squint/step-back tools were not visually obvious enough for beginners; only 3 levels was too few repetitions. Both addressed in Interaction 03.

Interaction 03 — UI aesthetic overhaul & level expansion

Stage: Polish & scope extension · Student: Charlie. S · Agent: Claude

Request. Make UI more "aesthetic" (retro panel-inspired). Highlight squint/step-back with icons so they jump out. Add more levels to reinforce learning through repetition.

What changed. Full CSS rewrite: warm studio palette, defined panel borders, green accent for tool buttons, gradient CTAs, entrance animations.
Squint / Step-back / Flip replaced with iconic tool buttons: SVG eye/shrink-arrows/flip-arrows inside green-accented pills with keyboard hints.
(Levels 4–6 were later redefined as new reference scenes — see Interaction 04.)

Interaction 04 — New reference scenes (Apple / Bird / Dragon)

Stage: Content expansion · Student: Charlie. S · Agent: Claude

Request. Keep the first reference (sphere+box) for Levels 1–3; after that, add three new, more interesting references instead of abstract variations. Level 4 = semi-realistic apple in colour with stand-out cool shadows (dark blue/green); Level 5 = a bird; final = a simple, fun cartoon dragon. Label post-L3 levels as Apple / Bird / Dragon (no numbers). Keep the block-in-values core.

What changed. scene.js now dispatches on cfg.kind: forms (L1–3), apple, bird, dragon. Each new scene is drawn from primitives (ellipses, circles, polygons) and still resolves to a 5-value map, so squint/pop/scoring are unchanged. Apple uses a cool blue-green shadow hue for realism; bird is blue with a warm beak; dragon is a friendly green with teal shadows. Scoring refactored to be region-data-driven (body/head/wing/leaf/stem…). Tabs relabelled L1 Grayscale / L2 Moving Light / L3 Colour / Apple / Bird / Dragon.

Validation. All 6 levels pass headless checks: full 1–5 value range, perfect = 100/win, blank = 0/lose, correct region sets per kind. Offline-safe.

Interaction 05 — UI layout rebalance (two-sided controls)

Stage: Interface readability · Student: Charlie. S · Agent: Claude

Request. Move interactable elements to two sides of the screen instead of only the right, and separate informational elements so the controls are easier to read and use.

What changed. The 2-column layout (canvas + single right sidebar) became a 3-column layout:
• Left — interactable painting tools: value palette, brush size, colour-mix (hue) slider.
• Center — the player canvas.
• Right — observation + scoring (informational, separated): reference image, KPIs (strokes left / used / AI score), painter's tool buttons (squint / step-back / flip), action buttons, and result + critique.
Small section-label dividers ("Painter's tools", "Actions") keep feedback clearly apart from controls. Swatches are now fluid so they fill the narrower left column. No JS changed — only structure + CSS, so behavior is identical.

Validation. JS syntax-checked; exhibition served at 1920×1080, all assets 200, no absolute/CDN paths. Interactable controls now sit on the left and right sides; reference/KPIs/result isolated on the right.

Interaction 06 — Tools & Actions moved to the left column

Stage: Interface readability (follow-up) · Student: Charlie. S · Agent: Claude

Request. Put Tools & Actions on the left with the palette; keep Score, Result, Reference on the right.

What changed. Refined the Interaction 05 layout: the left column now holds all interactable controls together — palette, brush, colour-mix, the painter's tool buttons (Squint / Step-back / Flip), and action buttons (Undo / Reset / Submit / Detail) in a second "Tools & Actions" card. The right column keeps only informational panels: Reference, Score (KPIs), and Result (critique), split into three separate cards. Left column widened to 268px. No JS changed.

Validation. Element IDs unique (1 each); columns in DOM order tools → canvas → side; exhibition serves offline with all assets 200.

Interaction 07 — UI clarity, pure palette, gentler 80% scoring

Stage: Interface clarity & game feel · Student: Charlie. S · Agent: Claude

Request. Obvious Submit at the bottom of the canvas (it wins the game); use empty space so tools need no scrolling; palette must be pure monochrome (was yellow-tinted); evaluation too harsh — 80% = passing with bands (0–50 needs improvement, 60 a little work, 70 decent, 80 good, 90 great, 95–100 perfect); reference ~70% of canvas.

What changed. • Palette now pure neutral gray (R=G=B) so value steps are unambiguous.
• Scoring: win = score ≥ 80 & budget > 0; the brittle "every region within 1 step" gate removed so 80%+ truly passes. Added grade bands printed in result + critique.
• Submit moved to a full-width big button at the bottom of the canvas — the most prominent element.
• Space / no-scroll: added a goal strip under the tabs; left tool buttons are now full-width, stacked, larger.
• Reference displays at 70% of the canvas (294px, square), centered as a thumbnail. System Design win/lose note updated.

Validation. JS syntax-checked; all assets 200; Submit confirmed inside the canvas column; perfect solve still 100/win, blank 0/lose, and 80%+ now passes even if one region is slightly off.

Interaction 08 — Result as a semi-opaque pop-up

Stage: Interface clarity & game feel · Student: Charlie. S · Agent: Claude

Request. The result should be a semi-opaque pop-up on Submit that covers the canvas and is a little larger than it.

What changed. Removed the inline right-side Result card. Pressing Submit now opens a centered modal (semi-opaque backdrop + blurred panel, 440px over the 420px canvas) showing the big score, grade, reason and full critique, with Keep Painting / Enter Detail Mode and a close button. Esc closes; the pop-up hides on reset.

Validation. JS syntax-checked; all nine pop-up element IDs present; no stale inline-result references.

Interaction 09 — Fair background scoring & colour palette

Stage: Scoring fairness & colour tooling · Student: Charlie. S · Agent: Claude

Request. The AI is never satisfied with the background (too dark/too light) even though value 2 is correct; and colour levels should have a colour palette.

What changed. • A headless harness proved the background is objectively value 2 everywhere — the fault was the scorer treating every unpainted cell as maximum error while the background is 44–74% of the canvas.
• Fair blank rule: an unpainted cell now scores as its distance from value 1 (the white canvas), so a correctly-read light background passes without hand-filling thousands of cells; a blank canvas still fails and dark blanks are still penalised.
• Colour palette: each colour level now offers named colour chips (e.g. Apple Red / Leaf Green / Stem Brown / Sky / Shadow Blue); the value swatches re-tint to the chosen colour. Scoring stays value-based.

Validation. Old-vs-new scoring table (blank 62/68 < 80, perfect 100, forms-perfect-blank-bg 89/82 pass); node --check clean; assets 200.
