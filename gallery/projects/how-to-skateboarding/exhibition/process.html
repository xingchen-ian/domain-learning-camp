Development Process

This page is a concise timeline of how the game was built through human–AI collaboration. The full chronological log is in agent-development-log.md.

Timeline

  • 2026-07-15 — Session 1

    Project Brief & Planning

    The student shared the game design brief (how-to-skateboarding.md) and the system graph. The AI agent read the brief, restated the core learning shift, core loop, main variables, and challenge presets, and proposed an implementation plan using Track A (plain HTML + CSS + Canvas 2D, no build step).

    Decision: Track A chosen because the game is a 2D side-view and does not need 3D or complex dependencies. This keeps the exhibition build simple — a static folder with no npm install required.

  • 2026-07-15 — Session 1

    Project Structure Created

    Created the directory layout required by the brief: exhibition/ for the final static website, source/ for editable source, and root-level files (development-brief.md, system-graph.png, README.md, THIRD_PARTY.md, submission-manifest.json).

  • 2026-07-15 — Session 1

    Core Game Loop Implemented

    The AI agent wrote game.js with clearly separated sections: Environment Data (level definitions, terrain functions), Player-Controlled Data (initial speed, core height), and System-Calculated Results (speed, energy, balance, position). The physics engine models friction, gravity, core efficiency on uphill, and balance loss on downhill. Three challenge presets were defined with different friction values, wheel types, terrains, and an energy pad in Level 3.

    Test result: Game loads in the browser, start screen displays terrain preview and speed slider, all three levels are selectable, win/lose conditions trigger correctly.

  • 2026-07-15 — Session 1

    Rendering & HUD

    Canvas rendering was implemented: sky gradient, parallax mountains, smooth terrain fill with grass, skateboard with wheels, stick-figure character with a color-coded core dot (green = good, yellow = warning, red = danger), and a HUD showing speed, energy, balance, core height vs. ideal core, slope indicator, and distance progress.

  • 2026-07-15 — Session 1

    Project Website Built

    Created index.html (game idea, domain knowledge, system design), game.html (playable game with touch controls), process.html (this page), and shared style.css. Navigation links use relative paths so the project works inside a Gallery subfolder.

Core Loop Validation

Observe → Judge → Act → Feedback → Adjust

  • Observe: The player sees the terrain ahead, their speed, energy, and core position.
  • Judge: "Is my core too high for this downhill? Do I have enough speed to climb the next hill?"
  • Act: Press ↑ or ↓ to shift the core height.
  • Read feedback: The core dot changes color, the balance bar drops or recovers, the board climbs or gets stuck.
  • Adjust: On the next attempt, the player picks a different initial speed or adjusts the core earlier.

AI Influence on Design Direction

  • The AI agent proposed using sin²(πt) smooth hills so terrain slopes are continuous (no sharp kinks that would make physics jitter).
  • The AI agent suggested separating uphill energy penalty (wrong core = more speed loss) from downhill balance penalty (wrong core = fall risk), matching the brief's description of expert vs. novice behavior.
  • The AI agent added a speed-amplifies-balance-error rule so that "too fast makes player fall down" from the brief is mechanically enforced.
  • The AI agent introduced a control factor per wheel type: ice recovers balance slowly (slippery), cake recovers quickly (grippy) — making each level feel distinct beyond just friction numbers.

Open Questions / Known Issues

  • Wind direction was mentioned in the brief's observe step but is not yet implemented. It may be added in a future iteration as a visual arrow that slightly affects balance.
  • The game has not yet been playtested by classmates. Balance tuning (friction values, balance recovery rates) may need adjustment after real playtesting.
  • Sound effects (skateboard movement, falling) are not yet added. The brief mentions sound as feedback but the first playable version prioritizes visual feedback.
