Development Process

A chronological log of design decisions, implementation milestones, and what changed between versions.

Timeline

Brief review & game idea

Stage 1 · planning

Read the development brief and the IMA system graph. Identified the three data roles (environment, player-controlled, calculated results) and the three difficulty presets (full algorithms / brief hints / no hints + timer).

Tech track decision

Stage 1 · planning

Chose Track B (locally vendored library) with three.js 0.160.0. Pinned the version, downloaded the ES module build into exhibition/assets/vendor/. Avoided npm/Vite to keep the final exhibition build a single static folder that runs from any HTTP server.

First playable build

Stage 2 · implementation

Built the 3D Rubik's cube as 27 mini-cubes with 6 face materials each. Implemented face turns via a temporary pivot object, smooth easing animation, and face highlight on selection. Added scramble, undo/redo, and the basic stats HUD (time, moves, % solved).

Three levels & hint system

Stage 2 · implementation

Wired up the three difficulty levels. Level 1 shows a full algorithm for the current CFOP-style stage. Level 2 shows a brief hint only. Level 3 hides hints and starts a 5-minute countdown. The hint card updates whenever a move is made.

Camera & input polish

Stage 2 · polish

Added custom orbit-style camera (drag to rotate, scroll to zoom) and click-to-select-face via raycasting. Keyboard shortcuts (R, L, U, D, F, B + arrows) for fast play.

Win / give-up / time-up overlays

Stage 2 · polish

Win detection compares all 54 stickers against each face's center color. Added a Give Up button and a Time Up handler for Level 3, both showing a final summary modal.

Website & submission prep

Stage 3 · packaging

Built the multi-page static site (index / game / process), wrote README and THIRD_PARTY, recorded this process page, and confirmed all internal links use relative paths so the project works inside gallery/projects/group-XX/.

Decisions & trade-offs

Decision

Custom camera instead of OrbitControls

OrbitControls could not be vendored at the same version of three.js. To stay fully offline, a small custom orbit camera was written by hand. Same effect, no dependency.

Trade-off

Approximate stage detection

The hint system estimates the solving stage from percent-of-stickers-correct, not by deep pattern recognition. This trades perfect accuracy for code a beginner can read.

Decision

Three levels instead of nine

The brief asked for "2-3 challenge presets when they are clearly defined". The system graph names three (Full Algorithms / Brief Hints / No Hints + Timer) so the game exposes those three.

Open issue

Scramble depth is fixed

The scramble applies 20 random face turns. A future iteration could vary the scramble depth by level and store best times per level in localStorage.

How to run

From the project root, run a static HTTP server, then open the URL.

# from the project root
cd exhibition
python -m http.server 8080
# then open http://localhost:8080

You can also use npx http-server exhibition -p 8080 if you have Node installed. The game will not work via file:// because the JS modules need a real HTTP origin.
