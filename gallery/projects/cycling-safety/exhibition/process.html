Development Process

This timeline records the human-AI collaboration that built the Cycling Safety game. It is based on agent-development-log.md and reflects the actual development path — including decisions, changes, and open questions.

Timeline

Session 1 — Project Setup

Reading the Brief and Planning

The AI agent read the domain-learning brief and identified the core learning shift: rushing causes falls; experts control speed and direction. An implementation plan was created covering the observe-judge-act-feedback-adjust loop, the three data categories (environment, player-controlled, system-calculated), and the four challenge presets.

Decision: Use Track A (no build step) — plain HTML, CSS, and Canvas 2D. This keeps the exhibition build simple and requires no npm or Node.js on the iMac.

Session 1 — Structure

Project Website and File Structure

Created the project directory structure following the brief: exhibition/ for the final static website, with index.html (home), game.html (playable game), process.html (this page), and shared assets/css/ and assets/js/.

Decision: All internal links use relative paths so the project works inside a Gallery subfolder.

Session 1 — Core Implementation

Building the Smallest Complete Game Loop

Implemented the game engine in game.js with Canvas 2D rendering. The first version included: a scrolling top-down road, player bike with speed/steering control, obstacle collision, stability system, and win/lose conditions. The code clearly separates Environment Data, Player-Controlled Data, and System-Calculated Results as required by the brief.

Key mechanic: Curves have a "safe speed" based on curve intensity and terrain grip. Exceeding it drops stability rapidly. This directly teaches the core learning shift.

Session 1 — Feedback System

Adding Visual and Audio Feedback

Added screen shake when taking curves too fast or riding rough terrain, skid marks on slippery surfaces, dust particles on gravel, speed lines at high velocity, and a red vignette when stability is critical. Audio uses the Web Audio API to generate wind sound (volume scales with speed), crash sound, skidding sound, and a win jingle — no external audio files needed.

Testing: Confirmed that feedback appears immediately for speed/terrain effects and with a slight delay for traction loss, matching the brief's feedback priority requirements.

Session 1 — Challenge Presets

Implementing Four Levels

Created four levels that differ through variables and relationships, not just visuals: Level 1 (pavement, gentle curves, sticks), Level 2 (gravel reduces grip, rocks crash instantly), Level 3 (hills affect momentum, sharp curves), Level 4 (slippery surface causes skidding, dense obstacles, tight curves). Each level has different terrain distributions, curve intensities, obstacle densities, and time limits.

Design choice: Rocks cause instant crash (teaching players to always avoid them), while sticks and branches reduce stability (graduated consequence). This creates a clear hierarchy of danger.

Session 1 — HUD and UI

Exposing Game Data on Screen

Built a HUD showing all required data: Speed (number + slow/moderate/fast label), Stability (bar with color: green/yellow/red + percentage), Terrain Type (pavement/gravel/slippery/rough), Distance to Destination (number + progress bar), Time Remaining (countdown, turns red when low), and Obstacle Alert (warning when an obstacle is in the player's path). Also added a lean indicator and low-stability warning flash.

Session 1 — Website Content

Building the Project Website

Created index.html with Game Idea, Domain Knowledge, and System Design sections. All content is based on the brief — no invented information. Navigation connects Home, Domain, System, Process, and Play pages. Styling is consistent across informational pages and the game.

Session 1 — Testing

Local Testing

Started a local HTTP server and tested navigation links, all four levels, win condition, all three lose conditions (crash, fall, timeout), pause/resume, and restart. Confirmed the game runs without internet access and no external dependencies.

Open question: Time limits and obstacle density may need tuning after playtesting with classmates. The current values are set to be challenging but achievable with careful play.

How AI Helped

Implementation The AI agent wrote the game engine, road generation, physics, rendering, and audio systems based on the design brief. All code is commented for a high-school student to understand.
Design Suggestions The AI suggested the graduated obstacle consequence system (rocks = crash, branches = major damage, sticks = minor damage) and the lean-into-curve mechanic to reinforce the domain learning.
Structure The AI created the file structure, README, development log, and manifest following the brief's requirements for Gallery compatibility.
Student Decisions The student chose the cycling domain based on personal experience, defined the core learning shift, and specified the four levels. Final tuning of difficulty and consent for public display are pending student confirmation.
