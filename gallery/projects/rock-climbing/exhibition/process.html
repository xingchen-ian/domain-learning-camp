Development Process

This page tracks how the Rock Climbing game was built through human-AI collaboration. It is based on the chronological development log and records both successes and failures honestly.

Session 1 — Project Setup

Read the brief and created the implementation plan

The AI agent read the full development brief and system graph. It restated the core learning shift (beginners think climbing = grab and pull; experts know it = read holds, position body, manage energy) and identified the key game variables: stamina, stability, hold types, grip force, and rest positions. The agent chose Track A (no build step) because the game is 2D and Canvas 2D is sufficient.

Session 1 — Project Setup

Created project structure and website skeleton

Set up the exhibition/ folder with index.html (home), game.html (playable game), process.html (this page), and assets/ (CSS + JavaScript). Created shared navigation and styling. Initialized agent-development-log.md with project metadata.

Session 1 — Core Implementation

Built the core game loop: observe → judge → act → feedback → adjust

Implemented the game engine in game.js with Canvas 2D. The engine separates data into three clearly labeled sections: Environment (holds, wall, route), Player-controlled (stamina, stability, position), and System-calculated results (height, shake, lean). The player clicks on highlighted reachable holds to climb. Each move goes through a grip meter that teaches grip force management.

Session 1 — Core Implementation

Implemented stamina, stability, and feedback systems

Stamina depletes with each move based on hold type, quality, grip force, and stability. Rest holds recover stamina with diminishing returns. Stability drops on wobbly holds and can be recovered with the "Adjust" action. Visual feedback includes arm shaking (red color + shake lines when stamina < 50%), body leaning (when stability < 40%), and chalk particle effects on successful moves.

Session 1 — Core Implementation

Created 3 challenge presets with different system variables

First Ascent: 14 holds, 40% jugs, 25% rests, 8% wobbly, large grip zone (58%). Crux Master: 18 holds, 28% crimps, 32% wobbly, small grip zone (52%). The Marathon: 22 holds, 1.15× stamina drain, 14% rests, longer gaps between holds. Each route changes actual game variables — not just visual decoration.

Session 1 — Polish & Testing

Added fall/win conditions, personal best tracking, and UI polish

Fall condition: stamina = 0 and not on a rest hold → fall animation → game over screen showing height reached. Win condition: reach the top hold → victory screen. Personal best tracked per-route in localStorage. Menu overlay with route selection, how-to-play guide, and best score display. Tested at 1920×1080 resolution.

Session 1 — Website

Built the project website with all required content

Home page covers game idea, domain knowledge (novice vs expert comparison), system design (core loop, data model, feedback, success/failure, challenge presets), and just-right ranges. Development process page (this page) mirrors the development log. Navigation connects Home, Domain, System, Process, and Play pages.

How AI Helped
  • Translated the design brief into a concrete implementation plan with clear variable names
  • Separated environment data, player-controlled data, and calculated results in the code
  • Designed the grip meter mechanic to teach "just-right" grip force (a key domain concept)
  • Generated deterministic route layouts using seeded random for consistent challenge presets
  • Implemented visual feedback systems (shaking, leaning, particles) from the brief's feedback priorities
Student Decisions
  • Chose Track A (no build step) — keeps the exhibition simple, no Node.js or npm required
  • Approved the grip meter as the core skill mechanic for teaching grip force management
  • Selected 3 routes with escalating difficulty: beginner, technical, endurance
Relationship to the Core Learning Shift

The game directly teaches the shift from "grab and pull" to "read, position, and manage." Players who rush up the wall without resting will run out of stamina and fall — exactly the novice mistake described in the brief. Players who read hold types, use rests, and control their grip force will climb higher. The feedback systems (shaking, leaning, grip results) make the cause-and-effect relationship visible and learnable.

Problems / Open Questions
  • The grip meter appears on every move — could be made optional or contextual in a future version
  • Wind conditions are mentioned in the brief but not yet implemented as a gameplay variable
  • No audio feedback yet — could add chalk sounds, grip sounds, and fatigue breathing
  • Foot placement is abstracted (previous hold = feet) rather than independently controlled
